# GeekeriOS

**An AI-native RevOps operations platform that turns Claude Desktop into a multi-agent command center with full observability and persistent memory.**

GeekeriOS connects [VoltAgent](https://github.com/VoltAgent/voltagent) (agent orchestration), [ruflo](https://github.com/ruvnet/ruflo) (cross-session vector memory), and [VoltOps](https://console.voltagent.dev) (real-time tracing) through the [Model Context Protocol](https://modelcontextprotocol.io). It runs entirely on your local machine — your API keys, your data, your disk.

```
Claude Desktop Chat ←stdio→ VoltAgent MCP Server (:3141)
                                    │
                  ┌─────────────────┼─────────────────┐
                  │                 │                  │
           Supervisor Agent    ruflo MCP         VoltOps Console
         (revops-orchestrator) (memory tools)   (console.voltagent.dev)
                  │
    ┌─────────────┼─────────────┬─────────────┐
    │             │             │             │
CRM Analyst  Data Researcher  Report Writer  Strategy Advisor


launchd (macOS scheduler)
  ├─ 7:00 AM MT, Mon–Fri → Briefing Runner → #geekerios-briefings
  ├─ 4:00 PM MT, Mon–Fri → Briefing Runner → #geekerios-briefings
  ├─ Every 30 min        → Meeting Prep    → #geekerios-briefings
  └─ 12:00 PM daily      → Token Keepalive → (silent)
         │
    ┌────┴────┬──────────┬──────────┬───────────┬──────────┐
  HubSpot  GCal     Gmail    Slack   Fireflies  GDrive
    API     API      API      API     GraphQL     API
```

You chat in Claude Desktop. The supervisor breaks complex requests into subtasks and delegates to specialist sub-agents (real Claude instances). ruflo stores patterns and findings that persist across sessions. VoltOps traces every delegation, tool call, and LLM response with timing and token costs. Scheduled briefings run automatically via macOS launchd, gathering live data from six services, tracking action items across runs, saving metric snapshots for weekly retros, and posting AI-synthesized summaries to Slack. Pre-meeting prep briefs arrive in Slack before external calls.

Everything is config-driven. Two JSON files define all agents, skills, and tools — editable from Claude Desktop's Chat tab using natural language.

---

## What It Does

- **Multi-agent orchestration** — A supervisor agent delegates to 4 specialists working in parallel. Pipeline audits, revenue forecasts, and client deliverables are generated by the right expert for each subtask.
- **Persistent memory** — ruflo's vector memory stores consulting frameworks, client findings, and learned patterns across sessions. `memory_search` retrieves relevant past work automatically.
- **Full observability** — VoltOps Console shows every agent delegation, tool call, LLM request/response, token usage, and latency in a visual execution trace. No black box.
- **Scheduled briefings** — Automated morning briefings (7 AM) and daily wraps (4 PM) posted to Slack, pulling live data from HubSpot, Google Calendar, Gmail, Slack, Fireflies, and Google Drive. Monday editions include a week preview and active docs; Friday editions include a week-over-week metrics retrospective with real deltas.
- **Meeting transcript integration** — Evening wraps pull today's meeting transcripts from Fireflies and surface decisions made and action items committed — not just "you had a meeting at 9 AM."
- **Action item tracking** — Evening briefings extract action items and persist them locally. Morning briefings surface open and overdue items from previous days. The loop closes automatically across runs.
- **Pre-meeting prep** — 30 minutes before external meetings, a prep brief arrives in Slack with attendee intel cross-referenced against HubSpot contacts and deals, plus suggested talking points.
- **Metric snapshots** — Daily pipeline metrics (deals open, pipeline value, tasks) are saved after each briefing. Friday retros show real week-over-week deltas instead of guessing.
- **Google Drive awareness** — Monday briefings surface recently modified documents across Drive, so you know what's been active.
- **Token health monitoring** — A daily keepalive keeps Google OAuth tokens alive, and the runner sends a Slack alert with re-auth instructions if a token expires.
- **Config-driven** — Add agents, update instructions, assign skills, and toggle tools by editing JSON files. No code changes for 90% of modifications.
- **Local-first** — Runs on your machine. Data stays on your disk. Traces persist to local SQLite.

## Quick Start

### 1. Clone and install

```bash
git clone https://github.com/YOUR_USERNAME/geekeriOS.git
cd geekeriOS/geekeri-volt
npm install
```

### 2. Configure your API key

From inside the `geekeri-volt/` directory:

```bash
cp .env.example .env
```

Edit `.env` and add your Anthropic API key:

```
ANTHROPIC_API_KEY=sk-ant-api03-your-key-here
```

### 3. Add the MCP server to Claude Desktop

#### 3a. Edit the config file

Open Claude Desktop and navigate to **Settings → Developer → Edit Config**. This opens `claude_desktop_config.json` in your editor. If you prefer to open it manually:

- **macOS:** `~/Library/Application Support/Claude/claude_desktop_config.json`
- **Windows:** `%APPDATA%\Claude\claude_desktop_config.json`
- **Windows (MSIX/Store install):** `%LOCALAPPDATA%\Packages\Claude_pzs8sxrjxfjjc\LocalCache\Roaming\Claude\claude_desktop_config.json`

Add the `geekeriOS` MCP server. Update the paths to match where you cloned the repo and where your Node 20+ binary lives:

```json
{
  "mcpServers": {
    "geekeriOS": {
      "command": "bash",
      "args": [
        "-c",
        "export PATH=/PATH/TO/NODE20/bin:$PATH; cd /PATH/TO/geekeriOS; npx -y ruflo@latest init >/dev/null 2>&1 || true; npx -y ruflo@latest daemon start >/dev/null 2>&1 || true; cd geekeri-volt; exec npx tsx src/index.ts"
      ]
    }
  }
}
```

The startup chain does four things in order: forces Node 20 onto the PATH, initializes ruflo in the project root (silent), starts the ruflo daemon (silent), then `cd`s into the `geekeri-volt/` subfolder where VoltAgent lives and launches the MCP server.

**macOS with nvm example:**

```json
{
  "mcpServers": {
    "geekeriOS": {
      "command": "bash",
      "args": [
        "-c",
        "export PATH=$HOME/.nvm/versions/node/v20.19.6/bin:$PATH; cd $HOME/Desktop/geekeriOS; npx -y ruflo@latest init >/dev/null 2>&1 || true; npx -y ruflo@latest daemon start >/dev/null 2>&1 || true; cd geekeri-volt; exec npx tsx src/index.ts"
      ]
    }
  }
}
```

**Windows example:**

On Windows, use `cmd /c` with `&&` chaining. Adjust paths to match your Node 20+ install location and where you cloned the repo:

```json
{
  "mcpServers": {
    "geekeriOS": {
      "command": "cmd",
      "args": [
        "/c",
        "set PATH=C:\\Users\\YOU\\.nvm\\versions\\node\\v20.19.6;%PATH% && cd /d C:\\Users\\YOU\\Desktop\\geekeriOS && npx -y ruflo@latest init >nul 2>&1 & npx -y ruflo@latest daemon start >nul 2>&1 & cd geekeri-volt && npx tsx src/index.ts"
      ]
    }
  }
}
```

If you installed Node 20+ system-wide (not via nvm-windows), replace the `set PATH=...` portion with your Node install path, e.g. `set PATH=C:\\Program Files\\nodejs\\;%PATH%`.

Save the file.

> **Important:** Do NOT use an external shell script file (`.sh` / `.bat`) — macOS Gatekeeper blocks downloaded scripts with "Operation not permitted" even after `chmod +x`. The inline `bash -c` approach avoids this entirely.

#### 3b. Grant folder access

Claude Desktop needs filesystem access to the geekeriOS directory so it can read and edit your config files (`agents.json`, `skills.json`) from the Chat tab.

1. Open Claude Desktop → go to the **Chat tab**
2. Click the **"+" button** at the bottom of the chat input
3. Select **"Add folder"** (or look for "Choose a folder" option)
4. Navigate to your geekeriOS project folder and select it
5. Claude Desktop will now have read/write access to `agents.json`, `skills.json`, and all other project files

This is what lets you say "Read my agents.json and add a new agent" directly in Chat.

#### 3c. Verify the MCP connector

1. Fully quit Claude Desktop (**Cmd+Q** on macOS, not just close the window)
2. Relaunch Claude Desktop
3. Open a new conversation in the **Chat tab**
4. Click the **"+" button** at the bottom of the chat input → select **"Connectors"**
5. You should see **"geekeriOS"** listed with a green status indicator
6. Click on it to see the available tools — you'll see ruflo memory tools (`memory_store`, `memory_search`, etc.) plus your built-in tools (`calculate_metrics`, `format_currency`)

If the connector shows red or doesn't appear:
- Check **Settings → Developer** for connection status and log links
- Open MCP logs at `~/Library/Logs/Claude/` (macOS) or `%APPDATA%\Claude\logs\` (Windows)
- Look for a file named `mcp-server-geekeriOS.log` for specific errors
- Common causes: wrong Node version (must be 20+), missing `npm install`, incorrect paths in the config

#### 3d. Optional — Add web connectors for business tools

The geekeriOS MCP server handles agent orchestration and memory. For connecting to business tools like your CRM, email, and calendar, add web connectors through Claude Desktop's built-in connector directory:

1. Go to **Settings → Connectors**
2. Browse the available integrations and connect the ones you use:
   - **HubSpot** — CRM data, deals, contacts, companies
   - **Salesforce** — pipeline, opportunities, accounts
   - **Gmail** — email search, drafts, sending
   - **Google Calendar** — events, scheduling, availability
   - **Slack** — messages, channels, search
   - **Notion** — pages, databases, knowledge bases
   - **Google Drive** — documents, spreadsheets, presentations
3. Click **"Connect"** for each and complete the OAuth authorization

These web connectors are available in both the **Chat tab** and **Cowork tab**. They work alongside the geekeriOS MCP server — for example, you can ask the orchestrator to pull HubSpot data via the web connector and then delegate analysis to the CRM Analyst sub-agent.

> **Note:** The geekeriOS MCP server (local stdio) only works in the **Chat tab**, not Cowork. Web connectors work in both. This is a [known Anthropic limitation](https://github.com/anthropics/claude-code/issues/20377).

### 4. Test it

In the Claude Desktop **Chat tab**, try:

> Run a pipeline audit for a B2B SaaS company with $2M ARR. They have 45 deals across 5 stages, average sales cycle of 60 days. Demo-to-proposal conversion is 40%, proposal-to-negotiation is 55%, negotiation-to-close is 35%. Identify the biggest bottleneck and recommend fixes.

You should see the orchestrator delegate to the CRM Analyst and Data Researcher, then pass results to the Report Writer.

### 5. View traces

Open **https://console.voltagent.dev** — it auto-connects to `localhost:3141` and displays the full execution trace.

## Configuration

### agents.json — Agent Definitions

Defines the supervisor and all sub-agents. Each agent has:

| Field | Purpose |
|---|---|
| `name` | Display name, used for delegation routing and VoltOps display |
| `purpose` | Short description the supervisor sees when choosing who to delegate to |
| `instructions` | Full prompt the agent uses when executing a task |
| `model` | Anthropic model string (e.g. `claude-sonnet-4-20250514`) |

The `supervisor` section has an additional `guidelines` array — rules injected into the supervisor's system message.

**Default agents:** CRM Analyst, Data Researcher, Report Writer, Strategy Advisor

### skills.json — Skills and Tools

**Skills** are methodology modules injected into agent instructions at startup:

| Field | Purpose |
|---|---|
| `id` | Unique identifier |
| `name` | Human-readable label |
| `appliesTo` | Array of agent names that receive this skill |
| `content` | Instruction text appended to the agent's prompt |

**Tools** are executable functions toggled per agent:

| Field | Purpose |
|---|---|
| `id` | Must match a key in the `builtInTools` map in `src/index.ts` |
| `appliesTo` | Which agents get access |
| `enabled` | true/false toggle |

**Default skills:** Pipeline Audit Methodology, Revenue Forecasting Model, Tech Stack Assessment, Client Deliverable Standards, CRM Data Quality Checklist

**Built-in tools:** `calculate_metrics` (11 RevOps formulas), `format_currency` (K/M/B notation)

## Scheduled Briefings

GeekeriOS includes a comprehensive automated briefing system that posts to Slack, pulling live data from six connected services with persistent state across runs.

### How it works

The briefing runner (`briefing-runner/`) is a standalone Node.js script scheduled via macOS `launchd`. On each run it:

1. Gathers live data from HubSpot, Google Calendar, Gmail, Slack, Fireflies (evening only), and Google Drive (Monday only)
2. Loads open action items from previous runs and metric snapshots for comparisons
3. Feeds all gathered data and state as context to the Anthropic API with a chief-of-staff persona prompt
4. Claude synthesizes the data into a warm, conversational briefing
5. For evening wraps, the runner parses Claude's structured action items and saves them for tomorrow's morning briefing
6. Saves a daily metric snapshot from HubSpot data for Friday retros
7. Posts the result to `#geekerios-briefings` in Slack via webhook

### Schedule

| Schedule | Script | What it does |
|---|---|---|
| 7:00 AM MT, Mon–Fri | `runner.ts morning` | Morning briefing — calendar, inbox (incl. WCP digest), pipeline, Slack, open action items |
| 7:00 AM MT, Monday | `runner.ts morning` | Monday edition — adds full week preview, active Google Drive docs, top 3 priorities |
| 4:00 PM MT, Mon–Fri | `runner.ts evening` | Daily wrap — what happened today, Fireflies meeting outcomes, action items, tomorrow preview |
| 4:00 PM MT, Friday | `runner.ts evening` | Friday edition — adds week-over-week metric deltas, wins/improvements retro |
| Every 30 min | `meeting-prep.ts` | Pre-meeting prep — attendee intel + HubSpot deals before external meetings |
| 12:00 PM daily | `keepalive.ts` | Google token keepalive — refreshes OAuth token to prevent 7-day expiry |

### Data sources

| Source | Auth method | What it pulls | When |
|---|---|---|---|
| HubSpot CRM | Private App token | Recent deals, pipeline stages, open tasks | Every briefing |
| Google Calendar | OAuth refresh token | Today's events or full week (Monday) | Every briefing |
| Gmail | OAuth refresh token | Unread/flagged, needs-reply, WCP daily digest | Every briefing |
| Slack | Bot User OAuth token | Recent mentions, active channel summaries | Every briefing |
| Fireflies | API key (GraphQL) | Meeting transcripts: decisions + action items | Evening wraps only |
| Google Drive | OAuth refresh token | Recently modified documents across Drive | Monday briefings only |

Each source is independent — if one fails or isn't configured, the briefing still runs with the available data and notes what's missing.

### Persistent state

The runner maintains two local data files in `briefing-runner/data/` that persist across runs:

**`action-items.json`** — Action items extracted from evening wraps. Each item has a source (email, hubspot, fireflies, etc.), text, creation date, due date, and status. Morning briefings surface open and overdue items. Items auto-mark as stale after 7 days. Completed items are pruned after 30 days.

**`metric-snapshots.json`** — Daily pipeline metrics (deals open, pipeline value, deals in negotiation, tasks open) saved from HubSpot data after each briefing run. Friday retros load this week's and last week's snapshots and show real week-over-week deltas. Snapshots are kept for 60 days.

### Pre-meeting prep

The meeting prep script (`meeting-prep.ts`) runs every 30 minutes via launchd. It checks Google Calendar for meetings starting in the next 45 minutes that include external attendees (filtered by your email address). For each qualifying meeting, it:

1. Looks up each attendee in HubSpot contacts (job title, company, lead status)
2. Finds associated deals for each attendee (deal name, stage, amount)
3. Sends the context to Claude for a 30-second prep brief
4. Posts the brief to `#geekerios-briefings` with attendee intel and suggested talking points

Internal-only meetings and all-day events are skipped. The script exits quickly when no meetings are upcoming — it's designed to run frequently with minimal overhead.

### Token health monitoring

Google OAuth "Testing" mode tokens expire after 7 days of inactivity. To prevent silent briefing degradation:

**Keepalive** (`keepalive.ts`) — Runs daily at noon (including weekends) via launchd. Refreshes the Google access token to keep the refresh token alive. If the token has already expired, it posts a re-auth guide to `#geekerios-briefings` with step-by-step instructions.

**Runner alerts** — If any briefing run detects a Google auth failure (`invalid_grant` or `unauthorized_client`), it sends a one-time Slack alert with fix instructions, then continues the briefing with whatever sources did work.

### Customizing

**Prompt templates** — Edit `briefing-runner/src/prompts.ts`. Each of the four briefing types (morning, monday, evening, friday) is a plain template string you can modify. The system prompt controls the overall persona and tone.

**Data sources** — Add or remove service modules in `briefing-runner/src/sources/`. Each module exports a single `gather*()` function that returns formatted text. The runner conditionally includes sources based on briefing type (e.g., Fireflies only in evening wraps, Google Drive only on Mondays).

**Schedule** — Edit the `Hour` and `Minute` values in the plist files under `briefing-runner/plists/`, then `launchctl unload` + `load` to apply.

**Action item behavior** — Edit thresholds in `action-tracker.ts`: stale after 7 days (configurable), pruned after 30 days. Items are deduplicated by partial text match to avoid duplicates across runs.

### Cost

Each briefing costs approximately $0.02–0.05 (Sonnet input/output tokens). Meeting prep costs ~$0.01–0.02 per meeting. At 10 briefings + 5 meeting preps per week, that's roughly $0.25–0.60/week.

The fastest way to modify GeekeriOS is through Claude Desktop Chat, since it has filesystem access.

### Add an agent

```
Read agents.json and add a new sub-agent called "Customer Success Analyst"
that specializes in churn analysis, NRR tracking, and customer health
scoring. Use model claude-sonnet-4-20250514. Save the file.
```

### Add a skill

```
Read skills.json and add a skill called "MEDDPICC Qualification" that
applies to CRM Analyst and Strategy Advisor. Content should cover Metrics,

```

```
Read skills.json and enable the calculate_metrics tool for Strategy Advisor. Save.
```

### Update the supervisor

```
Read agents.json and add a new guideline to the supervisor:
"Always check data quality before running pipeline analysis". Save.
```

### Review current configuration

```
Read both agents.json and skills.json, summarize all agents with
their assigned skills and enabled tools.
```

**After any edit:** Cmd+Q Claude Desktop and relaunch to apply changes.

### Adding new executable tools

New tools with custom logic require editing `src/index.ts`:

1. Add the tool to the `builtInTools` map using `createTool` with a Zod schema
2. Add a matching entry in `skills.json` with `enabled: true` and the `appliesTo` array

## ruflo — Memory Layer

ruflo provides cross-session vector memory via MCP tools. Based on extensive testing, these tools have verified working backends:

| Tool | What it does |
|---|---|
| `memory_store` | Save key-value data to persistent sql.js storage |
| `memory_search` | Vector search across stored patterns |
| `memory_retrieve` | Fetch a specific key |
| `memory_list` | List stored keys in a namespace |
| `hooks_session-start` | Begin a tracked session |
| `hooks_session-end` | End a session |
| `swarm_health` | Check coordinator status |
| `coordination_metrics` | Latency/throughput stats |

**Note:** ruflo's agent execution tools (`agent_spawn`, `task_create`, `hive-mind_init`, `workflow_run`) create metadata records but have no execution runtime in the MCP context. VoltAgent's native supervisor + sub-agent system handles actual agent orchestration. ruflo is used solely for its functional memory and session management.

## VoltOps Observability

Open **https://console.voltagent.dev** while the platform is running. It connects to `localhost:3141` via WebSocket.

**What gets traced:**
- Every `delegate_task` call from supervisor → sub-agent
- Every sub-agent's LLM request and response
- Every ruflo memory tool call
- Token usage per request
- Response latency per step
- Full execution tree

Traces persist to `.voltagent/observability.db` (LibSQL/SQLite) and survive process restarts.

For remote persistence, add keys to `.env`:

```
VOLTAGENT_PUBLIC_KEY=pk_...
VOLTAGENT_SECRET_KEY=sk_...
```

Get keys at [console.voltagent.dev/tracing-setup](https://console.voltagent.dev/tracing-setup).

| `EBADENGINE` warnings about Node version | Claude Desktop defaults to Node 18. Ensure the startup chain includes `export PATH=/path/to/node20/bin:$PATH` |
| stdout pollution errors (`Unexpected token`) | All setup commands before `mcp start` must redirect both stdout and stderr: `>/dev/null 2>&1` |
| Shell script "Operation not permitted" | macOS Gatekeeper blocks external scripts. Use inline `bash -c` commands instead. |
| Port 3141 already in use | Check with `lsof -i :3141`. Kill the existing process or change the port in `src/index.ts`. |
| ruflo `agent_spawn` / `task_create` fails | These tools create phantom records with no execution runtime. Use VoltAgent sub-agents for actual orchestration. |
| VoltOps Console shows "disconnected" | Verify the dev server is running: `curl http://localhost:3141/health` |
| Briefing not posting to Slack | Check `tail -50 briefing-runner/logs/morning.log`. Common causes: expired Google token, missing Slack webhook URL. |
| Google `unauthorized_client` in briefings | Refresh token was generated with the wrong client ID. Redo the OAuth Playground flow with "Use your own OAuth credentials" checked. |
| Google `invalid_grant` | Refresh token expired (7-day testing mode limit). Redo OAuth Playground flow. The keepalive prevents this during normal operation. |
| Google Drive returning 403 | Missing `drive.metadata.readonly` scope. Re-authorize in OAuth Playground with all three scopes. |
| Fireflies returning empty data | Check that your API key is valid and you have transcripts from today. Fireflies data only appears in evening wraps. |
| Slack `not_allowed_token_type` | You're using a user token where a bot token is needed, or vice versa. The runner uses `xoxb-` bot tokens only. |
| Slack `missing_scope` | Add the required scopes (channels:read, channels:history, groups:read, groups:history) to your Slack app and reinstall. |
| Meeting prep not firing | Check `launchctl list \| grep meeting-prep`. The script runs every 30 min and exits silently when no external meetings are upcoming. |
| Action items not appearing in morning | The evening wrap must run first to extract and save items. Check `briefing-runner/data/action-items.json`. |
| `launchctl` schedules not firing | Run `launchctl list \| grep geekerios`. If not listed, reload the plists. Machine must be awake at the scheduled time (launchd runs missed jobs on wake). |

## Known Limitations

- **Cowork tab** does not support local stdio MCP servers ([Anthropic bug #20377](https://github.com/anthropics/claude-code/issues/20377)). Use the Chat tab.
- **Custom connectors** in Claude Desktop require HTTPS + OAuth. Local HTTP servers cannot be added as connectors.
- **Config changes require restart.** `agents.json` and `skills.json` are read at startup.
- **ruflo daemon instability.** The daemon can report "running" while actually stopped ([ruflo #1283](https://github.com/ruvnet/ruflo/issues/1283)). Use `hooks_session-start` as the reliable initialization path.
- **Briefing MCP connector auth.** Anthropic-managed MCP proxy servers (`*.mcp.claude.com`) don't accept standalone API tokens — the briefing runner uses direct REST APIs instead, which is more reliable for scheduled jobs.
- **Google OAuth tokens in testing mode** expire after 7 days of inactivity. The daily keepalive prevents this during normal operation. If you're away for 7+ days, the runner will alert you in Slack and you'll need to redo the 2-minute OAuth Playground flow.
- **Fireflies API rate limits.** Free and Pro plans are limited to 50 API calls/day. The runner makes 1-2 calls per evening briefing, well within limits.
- **Meeting prep requires HubSpot.** Attendee intel is cross-referenced against HubSpot contacts. Without HubSpot, the prep brief will still include meeting details and suggested talking points but without CRM context.
- **Action items are local.** Items are stored in `data/action-items.json` on disk. They don't sync to WCP or any external task manager (yet — WCP integration planned when API auth is available).
